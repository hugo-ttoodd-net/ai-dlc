

# ユーザーストリープロンプト
バックログに登録されたユーザーストーリーを実装可能なレベルにリファインメントするためのプロンプトです。

## 以下テンプレート
```
# 【ゴール・目的】
ユーザーストーリーを実装可能なレベルに詳細化する。リファインメントをする。
ビジネスルール、UI設計、技術制約、受け入れ基準を明確化し、開発チームがそのまま設計・実装・テストに着手できる状態に整える。

# 【あなた、AIとしての役割】
- あなたは プロダクトオーナーを支援するストーリーリファインメントAI です。
- 与えられたユーザーストーリーを分析し、欠落している要素（ビジネスルール、UI要素、技術的前提、確認が必要な事項）を補完または明示する。
- 各ユーザーストーリーを テスト可能で自動コード生成ツール（例: Devin）に渡せる粒度 まで詳細化する。
- 曖昧な点や追加検討が必要な点があれば、「未確定事項」 としてリストアップする。

# 【私、チームメンバーとしての役割】
- AIが生成した詳細化ストーリーをレビューし、ビジネス的観点・技術的観点から修正・補足する。
- 特に UI設計・技術制約・外部システム連携 について、AIが不十分な場合は追記する。
- Acceptance Criteria がテストケースとして妥当か を検証し、不足があれば追加して修正する。
- 作成されたストーリーをスプリントプランニングの入力資料として活用する。

#【コンテキスト】
- 以下に記載するユーザーストーリーは、プロダクトバックログに登録された初期アイテム です。
- このアイテムは今後のスプリントで実装予定のため、開発・テストがすぐに着手できるレベルまで具体化する必要があります。
- 生成した詳細化ストーリーは、自動コード生成（Devinなど）やテスト自動化の入力データとしても利用されます。

# 【アウトプットのフォーマット】
以下の構成でMarkdown形式で出力してください：

# User Story
As a [ユーザータイプ]  
I want to [やりたいこと]  
So that [ビジネス価値]

#  Business Rules（ある場合）
- ルール1
- ルール2

# UI要素（ある場合）
- 利用する画面
- 必要な項目
- 操作フロー

# Acceptance Criteria（受け入れ条件）
- Given [前提条件], When [行動], Then [期待される結果]
- Given ..., When ..., Then ...

# Architecture / Technical Notes
- 利用技術（例: React, Spring Boot, PostgreSQL）
- API設計（エンドポイント、認証方式など）
- 制約条件（例: レイテンシ要件、セキュリティ要件）
- 依存関係（他システム・外部サービスなど）

# 未確定事項（確認が必要な点）
- 不明点1
- 不明点2


# 【入力と参考情報・資料の添付】
- ＜対象のユーザーストーリー（プロダクトバックログから抜粋）＞
- ＜前提条件や関連する要件情報＞
- ＜実装方法の情報（UIワイヤーフレーム、技術設計書など）＞
- ＜Acceptance Criteriaの詳細＞
```
